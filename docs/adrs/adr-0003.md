# ADR-0003: Aggressive Nushell Version Tracking

## Status

✅ Accepted

## Context

nu_plugin_nw_ulid is a Nushell plugin that depends directly on `nu-plugin` and `nu-protocol`.
These crates are tightly coupled to a specific Nushell release and do not follow semver in the
traditional sense — each Nushell release can introduce breaking API changes in its plugin
interface.

The project's upgrade history illustrates the forces at play:

- **PR #30** upgraded from nu-plugin 0.106.1 to 0.109.1. This required bumping the MSRV from
  Rust 1.85.0 to 1.89.0 and migrating all Value construction from manual struct literals to
  the new helper methods (`Value::string()`, `Value::int()`, `Value::list()`, etc.).
- **Issue #38** exposed a transitive dependency break: the `interprocess` crate made a breaking
  API change that rendered `nu-plugin-core` 0.109.1 unbuildable from a clean `cargo install`.
  The only fix was to upgrade to nu-plugin 0.110.0, which pinned a compatible `interprocess`
  version.
- **PR #39** performed the 0.109.1 → 0.110.0 upgrade, resolving the transitive breakage.
- The 0.109.1 upgrade also triggered a **temporary Windows build failure** due to a
  `windows-sys` version mismatch in nu-protocol's dependency tree.

Delaying upgrades accumulates migration debt: API changes compound, transitive dependency
breaks block users from installing the plugin, and the project falls out of step with the
Nushell version that users actually run.

Two alternatives were considered and rejected:

- **Pin to a known-good version and upgrade only when forced.** Nushell does not offer
  long-term-support releases, so "known-good" is a moving target. Transitive dependency
  breaks (like the `interprocess` incident) can force an upgrade at an unpredictable time
  anyway, negating the stability benefit.
- **Support multiple Nushell versions simultaneously** via feature flags or compatibility
  shims. The maintenance cost would be disproportionate to the project's size and contributor
  base, and Nushell's plugin API changes are pervasive enough that shims would be fragile.

## Decision

We will track the latest stable Nushell release and upgrade `nu-plugin` and `nu-protocol`
promptly after each release.

Specifically:

- **Upgrade together.** All Nushell crate dependencies (`nu-plugin`, `nu-protocol`,
  `nu-test-support`) will always be pinned to the same Nushell release version.
- **Accept MSRV bumps.** When a new Nushell release raises the minimum supported Rust version,
  we will adopt that MSRV. The project's MSRV is dictated by upstream, not chosen
  independently.
- **Migrate immediately.** Any API changes in the new Nushell release will be addressed in the
  upgrade PR, not deferred.
- **Document breaking changes.** Each upgrade PR will describe what changed, what migrated, and
  any platform-specific impacts, so the history is available for future reference.
- **Update CI in lockstep.** The CI workflow's MSRV job and integration-test Nushell version
  will be updated in the same PR as the dependency bump.

## Consequences

- The plugin stays installable and compatible with the Nushell version users are most likely
  running, reducing support burden from version-mismatch issues.
- Transitive dependency breaks (like the `interprocess` incident) are resolved by upgrading
  rather than patching around them.
- Migration effort is spread across small, incremental upgrades rather than accumulating into
  large, risky jumps.
- Each Nushell release creates a maintenance task: update dependencies, migrate APIs, update
  CI, and update documentation. This is a recurring cost.
- The project's MSRV will advance at whatever pace Nushell dictates, which may exclude
  contributors or users on older Rust toolchains.
- Documentation (README, CONTRIBUTING, developer guides) must be updated in the same PR as the
  dependency bump to avoid version drift — a problem already observed when the 0.109.1 →
  0.110.0 upgrade updated `Cargo.toml` but left prose docs referencing 0.109.1.
- The step-by-step upgrade procedure is maintained in the "Upgrading the Nushell SDK" section of
  `CONTRIBUTING.md`, which lists the files and checks required for each upgrade PR.
- This strategy can be revisited if Nushell introduces a stability guarantee, an LTS release
  channel, or a semver-stable plugin API.
